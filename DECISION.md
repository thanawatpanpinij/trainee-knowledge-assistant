# Architecture Decisions

## Decision 1: Chose Prisma ORM over Drizzle ORM

### Context

ในการทำโปรเจกต์นี้มีเวลาจำกัดครับ (ประมาณ 120 ชั่วโมง) ผมเลยจำเป็นต้องเลือก ORM สำหรับ SQLite ที่สามารถเซ็ตอัปได้เร็วๆ เพื่อที่จะได้เอาเวลาไป focus กับฟีเจอร์หลักๆ ที่แอบยากกว่าอย่างพวกระบบ RAG หรือ Vector DB ครับ

### Alternatives Considered

จริงๆ ผมแอบเล็ง Drizzle ORM ไว้เหมือนกันครับ เห็นว่าตอนนี้กำลังเป็นที่นิยม และเด่นเรื่อง Performance, รวมถึงมีขนาดที่เบากว่า Prisma ด้วย

### Why Prisma

สุดท้ายผมก็ตัดสินใจเลือก Prisma แทน Drizzle ครับ ด้วยเหตุผลหลักคือผมเคยใช้งานมาก่อน เลยคิดว่าจะช่วยให้พัฒนาได้เร็วกว่าครับ และผมรู้สึกว่าในแง่ของ Developer Experience (DX) ก็ยังดีกว่าด้วยครับ เพราะโค้ดอ่านเข้าใจได้ง่ายกว่ามาก (เช่น `prisma.user.findMany()`) ไม่ต้องเขียน raw SQL เอง (เช่น `SELECT ... FROM ...`) ช่วยลดภาระทางความคิดไปได้เยอะ รวมถึงยังช่วยให้สามารถไป focus ส่วนอื่นๆ ที่ซับซ้อนกว่าได้ครับ
นอกจากนี้ Prisma ยังมีไฟล์ `schema.prisma` ที่เป็นเหมือนเอกสารโครงสร้างฐานข้อมูล (Single-source-of-truth) ในตัวด้วย ซึ่งผมคาดว่าอาจมีช่วยให้ผู้ตรวจสามารถทำความเข้าใจโครงสร้างโปรเจกต์ได้ง่ายขึ้นด้วยครับ

### Trade-offs

การเลือก Prisma ทำให้ผมต้องยอมแลกกับขนาดของ Bundle ที่ใหญ่ขึ้นครับ และ Performance อาจจะสู้ Drizzle ไม่ได้ รวมถึงต้องมานั่งทำ Singleton Client เพื่อแก้ปัญหา Hot Reload ของ Next.js ด้วย แต่อย่างไรก็ตาม ในบริบทของโปรเจกต์ขนาดเล็กและ run บน local เป็นหลัก เรื่อง Performance เลยไม่ได้ส่งผลกระทบชัดเจนนักเมื่อเทียบกับเวลาที่ประหยัดไปได้ครับ

## Decision 2: Chose Vercel AI SDK over Manual API Integration

### Context

โจทย์กำหนดให้สร้างหน้า Chat คุยกับ AI โดยมีฟีเจอร์นับ Token (Required) และถ้าทำ Streaming Response ได้จะได้คะแนนโบนัสด้วย ซึ่งปกติถ้าเขียนเองบน React ทั้งหมดเลย การจัดการพวก state ของแชตและการทำ stream ข้อความน่าจะวุ่นวายและต้องเขียนโค้ดเยอะมากครับ

### Alternatives Considered

พิจารณาว่าจะเขียน Fetch API ต่อกับ Provider ตรงๆ แล้วค่อยมาใช้ `useState` กับ `useEffect` เพื่อจัดการพวกประวัติข้อความ สถานะการโหลด และเอาข้อความที่ส่งมาทีละก้อน (Chunk) มาต่อกันโชว์บนหน้าจอเองครับ ซึ่งคาดว่าน่าจะต้องใช้ระยะเวลาในการเรียนรู้ประมาณหนึ่งเลยครับ

### Why Vercel AI SDK

ผมตัดสินใจเลือก Vercel AI SDK เพราะมันออกแบบมาคู่กับ Next.js เลยครับ ฝั่งหน้าบ้านก็มี hook อย่าง `useChat` ที่ช่วยจัดการ State ให้อัตโนมัติ ไม่ต้องมาปวดหัวเรื่องการ sync UI เอง ส่วนฝั่งหลังบ้าน (Backend) ก็มีฟังก์ชัน `streamText` ที่ช่วยทำเรื่อง Streaming ให้ง่ายๆ แถมยังดึงพวกข้อมูล Metadata อย่าง Token Usage มาใช้ได้ง่ายและเป็นมาตรฐานเดียวกันทั้งหมดครับ

### Trade-offs

การนำ SDK เข้ามาก็เป็นการเพิ่ม Dependency เข้ามาในระบบค และแอบเป็น Black Box พอสมควร หรือพูดง่ายๆ ก็คือเราไม่เห็นกลไก หรือการทำงานลึกๆ ของมันเลยครับ ถ้าหากอยากจะปรับแต่ง (Customize) เองลึกๆ อาจจะควบคุมได้ยาก รวมถึงต้องใช้เวลาทำความเข้าใจ Docs ที่อัปเดตบ่อยด้วย แต่ถ้าเทียบกับความเสถียรและเวลาที่ประหยัดไปได้ตอนทำ Streaming ก็ถือว่าคุ้มค่าและตอบโจทย์เวลาที่มีจำกัดมากๆ ครับ

## Decision 3: Auth Strategy: JWT with `jose` on Proxy (Next.js 16)

### Context

ระบบต้องการ Protected Routes (สำหรับหน้า `/chat` และ `/upload`) เพื่อให้เข้าใช้งานได้เฉพาะคนที่ยืนยันตัวตนแล้วเท่านั้น ซึ่ง Next.js 16 (App Router) ดักจับ Request พวกนี้ผ่านไฟล์ `src/proxy.ts` (หรือ middleware เดิม) ที่ทำงานอยู่บน Edge Runtime ครับ

### Alternatives Considered

ปกติเวลาทำ Frontend ทั่วไปผมจะคุ้นเคยกับการใช้ `jsonwebtoken` (JWT) ครับ หรือไม่ก็เป็นท่า Server-side Session ที่เก็บสถานะล็อกอินไว้ใน Database ไปเลย

### Why `jose` with HTTP-Only Cookie

เรื่องนี้เป็นเรื่องที่ผมเพิ่งได้เรียนรู้จากการทำโปรเจกต์ครั้งนี้เลยครับ ปกติผมใช้แต่ `jsonwebtoken` แต่เพิ่งทราบว่ามันพึ่งพา Module ของ Node.js (เช่น `crypto`) ซึ่งพอมันมารันบน Edge Runtime ของ Next.js มันจะพัง ผมเลยต้องเปลี่ยนมาใช้ Library อย่าง `jose` ที่เขียนขึ้นด้วย Web Crypto API เพื่อให้รันบน Edge Runtime ได้แทน ส่วนตัว Token ผมเลือกเก็บใน HTTP-Only Cookie เพื่อความปลอดภัยและป้องกันโจมตีแบบ XSS ครับ การใช้ JWT แบบ Stateless ยังช่วยให้ระบบไม่ต้องไปดึงฐานข้อมูลบ่อยๆ ในทุก Request ด้วย

### Trade-offs

ท่านี้ทำให้ผมต้องยอมแลกกับการไปนั่งเรียนรู้วิธีเขียน `jose` ใหม่นิดหน่อยครับ เพราะต่างจาก `jsonwebtoken` (เช่น ต้องมี `TextEncoder` แปลง key) และข้อเสียอีกอย่างของ JWT แบบ Stateless คือถ้าเราอยากจะยกเลิกสิทธิ์ (Revoke) กลางคัน จะทำได้ยากกว่าระบบ Session หากไม่เขียนระบบ Blacklist เพิ่มใน Database ซึ่งก็จะลด Performance ลงไปนิดนึงครับ
